# 受け入れ基準との照合ガイド

検証の最後に、受け入れ基準の各項目について、実際に検証した証拠を記録する。

## 照合の例

- 「ILoggerでログ出力するとCloudWatch Logsに振り分けられる」→ E2Eテストで検証済み
- 「xUnit 単体テストが通過する」→ 単体テスト28件全通過

## 重要な制約

⚠️ **重要**: 受け入れ基準の中に E2E テストでしか検証できない項目があるにも関わらず、E2E テストが未実施の場合は、検証を完了にしてはならない。該当項目を `NOT_VERIFIED` として記録し、ユーザーに報告すること。

### 🚨 E2Eテスト未実施時の厳格な対応ルール

**「環境制約」はE2Eテストをスキップする許容理由にならない。**

テスト戦略にE2Eが含まれる場合、E2Eテストが未実施のまま検証を完了とすることは一切認められない。以下のフローに従うこと：

#### E2Eテスト未実施時の対応フロー

```
1. E2Eテストが実行できない理由を具体的に特定・記録する
2. ユーザーに状況を報告し、以下の選択肢を提示する:
   a) 環境準備を行う（具体的な手順と必要なツール一覧を提示）
   b) テスト戦略を変更する（E2Eの範囲縮小・代替手段の提案）
   c) テスト戦略からE2Eを除外する（ユーザーの明示的な承認が必須）
3. ユーザーの決定を待ち、決定内容に従って対応する
4. いかなる場合もエージェントが独断でE2Eスキップを決定してはならない
```

#### 許容されないパターン

| パターン | 理由 |
|----------|------|
| 「Docker未インストールのためE2Eスキップ」 | ユーザーにDocker導入を依頼すべき |
| 「CI環境の制約でE2E省略」 | CI環境の設定変更をユーザーに提案すべき |
| 「時間的制約によりE2E後回し」 | E2E完了まで検証は完了にできない |
| 「E2E環境構築が複雑なためスキップ」 | 構築手順をユーザーに提示して判断を仰ぐべき |
