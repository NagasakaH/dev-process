# 体系的デバッグスキル

ランダムな修正は時間を浪費し新たなバグを生みます。根本原因を常に特定してから修正を試みてください。症状の修正は失敗です。

## 主要機能

### 4つのフェーズ

1. **Phase 1: 根本原因調査** - エラーメッセージ精読、再現、最近の変更確認、証拠収集
2. **Phase 2: パターン分析** - 動作例との比較、差分特定
3. **Phase 3: 仮説と検証** - 1つの仮説、最小限のテスト
4. **Phase 4: 実装** - 失敗テスト作成、単一修正、検証

## 鉄則

```
根本原因調査なしに修正を行わない
```

3回以上修正が失敗したら、アーキテクチャを疑う。

## 使用タイミング

- テスト失敗
- 本番バグ
- 予期しない動作
- パフォーマンス問題
- ビルド失敗

## レッドフラグ

- 「とりあえず修正、後で調査」
- 「Xを変えてみよう」
- 調査前に解決策を提案
- 「もう1回修正を試す」（2回以上失敗後）

## 関連スキル

- `test-driven-development` - Phase 4の失敗テスト作成
- `verification-before-completion` - 修正成功の確認
- `investigation` - 複数コンポーネントにまたがる調査
