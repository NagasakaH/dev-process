# MR/PR作成スキル（create-mr-pr）

Draft MR/PRを作成し、テンプレート付きチェックリストをdescriptionに設定するスキルです。

## 2つのモード

| モード | 対象リポ | 運用 | 用途 |
|--------|----------|------|------|
| **DR** | dev-process | close（マージしない） | 設計レビュー |
| **Code** | 各submodule | merge | コードレビュー |

## 入力

| 入力 | 必須 | 説明 |
|------|------|------|
| モード | ✅ | `dr` または `code` |
| ブランチ名 | ✅ | `git branch --show-current` で取得 |
| ベースブランチ | ✅ | マージ先（通常 main/master） |
| チケットID | 推奨 | ブランチ名やコミットメッセージから推測 |

## ワークフロー

```
1. プラットフォーム検出（GitHub/GitLab）
2. モード判定 → テンプレート選択
3. テンプレートファイル読み込み（code-template.md / dr-template.md）
4. テンプレートの各セクションに実際のコンテンツを埋め込み
5. テンプレート構造検証（後述）
6. 検証通過後、draft MR/PR作成（API送信）
7. 作成後body確認: APIでMR/PRのbodyを再取得し、意図した内容と一致するか検証
8. [Codeモード] 統合MR/PR要否判定 → 必要なら作成
```

### テンプレート強制適用ルール

MR/PR作成時は**必ず**以下のフローに従うこと。テンプレートなしでの直接作成は禁止。

#### Step 1: テンプレートファイル読み込み

- Codeモード → `references/code-template.md` のテンプレート構造を使用
- DRモード → `references/dr-template.md` のテンプレート構造を使用

#### Step 2: セクション埋め込み

テンプレートの各プレースホルダ（`{タイトル}`, `{変更の目的...}` 等）を実際の情報に置換する。

#### Step 3: 構造検証（API送信前の必須ゲート）

以下を**全て**満たさない限りAPI送信しないこと:

1. **必須セクション存在チェック**: テンプレートの必須見出しがテキスト検索で存在することを確認
   - **Codeモード**: `### 概要`, `### 変更内容`, `### AI自動チェック`, `### AI+人間チェック`
   - **DRモード**: `### 概要`, `### 設計資料`, `### 修正対象リポジトリ`, `### Acceptance Criteria テストマッピング`, `### AI自動チェック`, `### AI+人間チェック`
   - 各見出し直後のセクション本文が空（空行のみ or プレースホルダのみ）でないこと

2. **AI自動チェック根拠検証**:
   - **チェック済み（`- [x]`）項目**: 具体的な根拠が必須
     - チェック済み項目の直後に `根拠:` 行が存在すること
     - `根拠:` の値がプレースホルダー（`{...}`）でないこと
     - `根拠:` の値が以下の**無効パターン**に該当しないこと:
       - 「OK」「確認済み」「問題なし」「テスト実行済み」など、具体的なコマンド・結果・証拠を含まない記述
     - ✅ **有効な根拠の例**:
       - `根拠: npm test → 247テスト全通過 (3.2s)`
       - `根拠: git diff --stat で変更ファイル一覧を確認、不要ファイルなし`
       - `根拠: eslint src/ → 0 errors, 0 warnings`
       - `根拠: CI pipeline #1234 passed (URL: https://...)`
     - ❌ **無効な根拠の例**:
       - `根拠: OK`
       - `根拠: 確認済み`
       - `根拠: テスト実行済み`
       - `根拠: 問題なし`
       - （共通点: 具体的なコマンド名・実行結果・数値・URLなどの客観的証拠がない）
   - **未チェック（`- [ ]`）項目で対象外の場合**: 対象外の理由を記載必須
     - 未チェック項目に `根拠:` 行も対象外理由も記載されていない場合は検証エラーとする
     - 対象外理由は項目テキスト内に `（対象外: {理由}）` の形式で付記すること
     - ✅ **有効な対象外理由の例**:
       - `- [ ] テスト全パス（対象外: ドキュメントのみの変更のためテスト対象なし）`
       - `- [ ] E2Eテスト（対象外: テスト戦略にE2Eが含まれていない）`
       - `- [ ] CI/パイプライン成功（対象外: CI未構成のリポジトリ）`
     - ❌ **無効な未チェック項目**: 理由なしの未チェック状態
       - `- [ ] テスト全パス`（対象外理由がなく、根拠もない）

3. **チェックリスト形式検証**: `- [ ]` または `- [x]` 形式が正しく使用されていること

#### Step 4: 作成後body確認

API送信後、作成されたMR/PRのbodyをAPIで再取得し、以下を確認:

- テンプレートの全セクションが意図通り含まれているか
- マークダウン構造が崩れていないか
- AI自動チェック項目と根拠が正しく反映されているか

**検証失敗時のフロー:**

1. MR/PRを**draftのまま保持**（削除しない）
2. 検証失敗の内容をユーザーにエラー報告（どのセクションが欠落/破損しているか具体的に提示）
3. 手動修正を要求: 修正すべき箇所と修正案をユーザーに提示し、承認を得てからdescriptionを更新・再検証する

📖 プラットフォーム検出: [references/platform-detection.md](references/platform-detection.md)

## DRモード

設計レビュー用MR/PRをdev-processリポに作成。

テンプレート内容:
- 設計概要 + 設計資料リンク
- ACテストマッピングテーブル（必須）
- 修正対象リポジトリ合意テーブル（必須）
- テスト戦略チェックリスト（拡充版）

📖 DRテンプレート詳細: [references/dr-template.md](references/dr-template.md)

## Codeモード

コードレビュー用MR/PRを各submodule（`submodules/editable/`配下）のリポに作成。

テンプレート内容:
- AI自動チェック（7項目）
- AI+人間チェック（8項目）

📖 Codeテンプレート詳細: [references/code-template.md](references/code-template.md)

### 統合MR/PR

複数submodule修正時、またはクロスリポ結合テストがある場合、dev-processリポに統合MR/PRを追加作成。

📖 統合MR/PR手順: [references/integration-mr-pr.md](references/integration-mr-pr.md)

## レッドフラグ

**絶対にしない:**
- draft以外でMR/PRを作成する
- テンプレートのチェックリストを省略する
- ACテストマッピングテーブルを空のまま作成する（DRモード）
- 修正対象リポジトリ合意テーブルを省略する（DRモード）
- **テンプレートを読み込まずにMR/PRを作成する（テンプレートなしでの直接作成禁止）**
- **AI自動チェック項目を根拠なしで「OK」にする（具体的な根拠の記載が必須）**
- **未チェック項目を対象外理由なしで放置する（対象外の場合は理由の記載が必須）**
- **構造検証をスキップしてAPI送信する**
- **作成後のbody確認を省略する**

## 完了報告

| モード | 報告内容 |
|--------|----------|
| DR | MR/PR URL、テンプレート設定完了 |
| Code | 各submodule MR/PR URL（+ 統合MR/PR URL） |
