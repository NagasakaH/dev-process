# DRモード テンプレート

設計レビュー用MR/PRのdescriptionテンプレート。

## テンプレート構造

```markdown
## 設計レビュー: {タイトル}

### 概要

{設計の目的・背景・スコープの簡潔な説明}

### 設計資料

- [詳細設計書](docs/{target_repo}/design/design-document.md)
- [API設計](docs/{target_repo}/design/api-design.md)
- [データ構造設計](docs/{target_repo}/design/data-structure.md)
- [処理フロー設計](docs/{target_repo}/design/processing-flow.md)
- [テスト計画](docs/{target_repo}/design/test-plan.md)

> 実際に存在するファイルのみリンクする。ファイル名は設計成果物に合わせて調整。

### 修正対象リポジトリ

| リポジトリ | 修正可否 | 理由 |
|---|---|---|
| {repo_1} | ✅ 修正可 | {主要な変更対象} |
| {repo_2} | ❌ 修正不可 | {理由} |
| dev-process | ✅ 修正可 | 設計・プロセス資料 |

- [ ] ⬜ レビュアー確認済み

### Acceptance Criteria テストマッピング

| # | Acceptance Criteria | テスト種別 | テスト内容 | 検証方法 | 状態 |
|---|---|---|---|---|---|
| 1 | {criteria_1} | 単体/結合/E2E | {具体的テスト内容} | {検証方法} | ⬜ |
| 2 | {criteria_2} | {種別} | {具体的テスト内容} | {検証方法} | ⬜ |

> ACの各項目に対して必ず1行以上のテストマッピングを記載すること。空行は許可しない。

### テスト環境準備状況

テスト計画の実行に必要なリソースと、現在の準備状況。

| # | 必要リソース | カテゴリ | 状態 | 備考 |
|---|---|---|---|---|
| 1 | {ランタイム/SDK名 バージョン} | ランタイム | ✅ 準備済 / ❌ 不足 | {インストール状況} |
| 2 | {テストフレームワーク名} | テストツール | ✅ 準備済 / ❌ 不足 | {バージョン等} |
| 3 | {DB種別} | データベース | ✅ 準備済 / ❌ 不足 | {ローカル/Docker等} |
| 4 | Docker / docker-compose | コンテナ | ✅ 準備済 / ❌ 不足 / N/A | {dockerデーモン起動可否} |
| 5 | {コンテナイメージ名:tag} | コンテナイメージ | ✅ pull可 / ❌ 不可 / N/A | {レジストリアクセス可否} |
| 6 | {認証情報/.env等} | 設定・認証 | ✅ 準備済 / ❌ 不足 | {マスク済み状態記載} |
| 7 | {E2Eツール名} | E2Eテスト | ✅ 準備済 / ❌ 不足 / N/A | {必要な場合のみ} |
| 8 | {サンプルデータ} | テストデータ | ✅ 準備済 / ❌ 不足 / N/A | {フィクスチャ等} |

> 不足リソースがある場合、レビュアーはリソース提供方法をコメントで指示すること。
> 全リソースが ✅ になるまで実装フェーズに進まない。

### AI自動チェック

> ⚠️ **根拠記載ルール**:
> - **`- [x]` 項目（チェック済み）**: 根拠の記載が必須
> - **`- [ ]` 項目（対象外）**: 対象外の理由を項目テキストに `（対象外: {理由}）` の形式で必ず付記
>
> 根拠には以下のいずれか1つ以上を含めること:
> - **テスト結果**: 検証コマンドの実行結果
> - **検証方法**: 確認に使用した手順やツール
> - **確認した証拠**: ファイルパス、レビュー結果URL、ログ等の客観的証拠
>
> ✅ 有効な根拠: `ls docs/repo/design/ → 5ファイル存在確認` / `review-design 実行 → PASS (2024-01-15)` / `wc -l test-plan.md → 120行`
> ❌ 無効な根拠: `OK` / `確認済み` / `テスト実行済み` / `問題なし`（具体的コマンド・結果・証拠がない）
>
> **対象外の例**:
> - `- [ ] UML図の整合性（対象外: 本設計にUML図を含まない）`
> - `- [ ] API定義の形式的正しさ（対象外: API変更を含まない設計）`

- [ ] 設計ドキュメント完全性（必須セクション存在）
  - 根拠: {確認したファイルパスと存在するセクション一覧を記載}
- [ ] UML図の整合性（参照先存在・構文エラーなし）
  - 根拠: {検証したUMLファイルと構文チェック結果を記載}
- [ ] review-designスキルによるAIレビュー完了
  - 根拠: {レビュー実行日時と結果サマリーを記載}
- [ ] API定義の形式的正しさ（エンドポイント・型定義）
  - 根拠: {検証したAPIファイルと形式チェック結果を記載}
- [ ] テスト計画ドキュメント存在
  - 根拠: {テスト計画ファイルパスと主要セクションを記載}
- [ ] AC↔テストケースマッピング定義あり
  - 根拠: {マッピングテーブルの行数とカバーしたAC番号を記載}
- [ ] テスト種別（単体/結合/E2E）定義あり
  - 根拠: {定義されたテスト種別と件数を記載}

### AI+人間チェック

**設計妥当性:**
- [ ] 要件(acceptance_criteria)との整合性
- [ ] 技術的実現可能性
- [ ] セキュリティ設計の妥当性
- [ ] パフォーマンス影響の考慮
- [ ] API設計の妥当性（命名・互換性）
- [ ] データ構造の妥当性（正規化・インデックス）
- [ ] エラーハンドリング・リトライ設計
- [ ] 既存システムとの統合影響

**テスト戦略:**
- [ ] テスト種別（単体/結合/E2E）の選定が適切
- [ ] テストピラミッドのバランスが妥当
- [ ] 各acceptance_criteriaに対応するテストケースが存在
- [ ] 正常系テストケースの網羅性
- [ ] 異常系・エッジケースのテストケース定義
- [ ] 境界値テストの考慮

**テスト品質:**
- [ ] テストの独立性（他テストに依存しない）
- [ ] テストデータの妥当性
- [ ] モック/スタブの使用範囲が適切

**目的達成の検証根拠:**
- [ ] 各テストが「何を証明するか」が明確
  - 記載要件: テスト名と証明対象の対応表を記載すること
- [ ] テスト結果から目的達成を判断できる基準が定義
  - 記載要件: 合否判定の定量基準（例: カバレッジ○○%以上、レスポンス○○ms以下）を記載すること
- [ ] 非機能要件（パフォーマンス/セキュリティ）のテスト方法定義
  - 記載要件: 使用ツール、測定方法、目標値を記載すること
```

## テンプレート生成ルール

1. **ACテストマッピング**: 設計書・テスト計画から自動生成。各ACに最低1テスト。
2. **テスト環境準備状況**: テスト計画で必要なリソースを洗い出し、現在の実行環境で利用可能か確認。不足リソースは ❌ としてマーク。
3. **修正対象リポ**: `submodules/editable/` 配下のリポをリストアップし修正可、`submodules/readonly/` 配下を修正不可として列挙。
4. **設計資料リンク**: `docs/{target_repo}/design/` 配下の実ファイルをスキャンしてリンク生成。
5. **チェックボックス**: 全て未チェック状態で作成。AIが自動チェック項目を検証後に更新。
6. **AI自動チェック根拠**: チェック済み（`- [x]`）項目は根拠の記載が必須。未チェック（`- [ ]`）項目で対象外の場合は対象外理由の記載が必須。根拠には具体的なコマンド名・実行結果・数値・URL等の客観的証拠を含めること。「OK」「確認済み」等の抽象的記述のみの根拠は構造検証で却下される。

## DR完了後の運用

- 人間がMR/PR上でレビュー → 指摘があれば設計修正
- レビュー承認後、MR/PRを**close**（マージしない）
- design_review チェックポイントで承認記録
