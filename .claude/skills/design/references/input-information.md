# 入力情報の詳細

## 1. 要件（必須）

設計の基礎となる要件情報：

- **機能要件**: ユーザーが実行できる操作、出力形式等
- **非機能要件**: 応答時間、スループット、セキュリティ要件等
- **受入基準**: 設計が満たすべき完了条件

## 2. 調査結果（必須）

事前調査で得られた知見：

- **調査サマリー**: アーキテクチャ調査結果の要約
- **重要な発見**: 既存コードの構造、テストカバレッジ等
- **リスク**: 特定されたリスク事項
- **調査ドキュメント**: docs/{target}/investigation/ 配下のファイル

調査ドキュメントの典型的な構成：

```
docs/
└── {target}/
    └── investigation/
        ├── 01_architecture.md
        ├── 02_data-structure.md
        ├── 03_dependencies.md
        ├── 04_existing-patterns.md
        ├── 05_integration-points.md
        └── 06_risks-and-constraints.md
```

## 3. 設計制約・決定事項（任意）

ブレインストーミングやチーム討議で決定された方針：

- **技術的決定**: 認証方式、データストア選定等
- **深掘り要件**: 追加で明確化された要件
- **制約条件**: 技術的・ビジネス的制約

## 4. 設計ドキュメント（任意）

既存の設計ドキュメントがある場合は、設計セクションも更新します。

## 入力情報の活用方法

### 要件の参照

機能要件・非機能要件を設計の基準として活用します：

- **機能要件** → インターフェース/API設計、データ構造設計の入力
- **非機能要件** → 実装方針決定、テスト計画の入力

### 調査結果の参照

調査で得られた知見を設計に反映します：

- **重要な発見** → 既存コードとの整合性を考慮
- **リスク** → リスク軽減策を設計に組み込む

### 設計制約・決定事項の参照

チームで決定された方針を設計に反映します：

- **技術的決定** → 技術選定の根拠として使用
- **深掘り要件** → 要件の詳細化
- 設計が要件を満たしているかの検証基準としても使用
